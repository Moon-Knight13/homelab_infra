homelab_infra — Engineering Overview

A technical read on how the platform is built: the layered architecture, the tooling at each layer, the phased build plan, and the quality gates every change clears before it merges. For the mission and capabilities in plainer language, see the platform briefing.

Scope: pattern-level engineering — no live deployment details (addresses, hosts, secrets). The reference stack is a validation target; the pattern is hardware-agnostic and driven entirely from inventory data.

Architecture

Six layers, each built on the one below

Every layer depends only on the layers beneath it — so a rebuild can never deadlock, and independently-built pieces stay compatible because the contracts between layers are fixed. Logic is generic; all deployment specifics live in inventory data.

L0Power & provisioningunattended network boot + installPXE · TFTP · HTTP · Redfish/IPMI
L1Virtualisationheadless hypervisor hosts + VM/LXC profilesProxmox VE
L2Network policyfirewall, VLAN segmentation, default-denypfSense · policy-as-code
L3HardeningCIS benchmark + scheduled drift scansansible-lockdown CIS
L4Platform servicesidentity, secrets, container mgmt, observabilityAuthentik · Vault · Arcane · Wazuh
L5Workloadsservices onboarded via readiness manifestsGHCR digests · Podman/Docker

Everything is defined in Ansible — idempotent, lint- and syntax-clean by contract. Network facts live in a single machine-readable policy source, so firewall rules, virtualisation checks, and service placement are all generated from it and cannot silently disagree.

Topology

Where the stack sits across the security zones

A pattern-level view: a default-deny firewall in front of a segmented virtual network on the hypervisor, each zone isolated with explicit allow-lists only. Zones are shown by role — no addresses, VLAN numbers, or hostnames.

Quality gates

Every change runs the same pipeline

Change lands as a pull request; automated gates must pass before merge; an administrator applies the merged branch from a controlled workstation. The platform never reconfigures itself and CI holds no keys to it.

lintansible-lint + --syntax-check on every role and playbook
policyrendered firewall rules match the policy source; a disallowed flow is blocked
secretsgitleaks + custom deny-list scan (IPs, topology, identity data)
CIS auditpost-audit gate blocks merge when Level 1 controls fail
image scanTrivy — fixable Critical/High blocks deploy; waivers to the risk register
check-modestable checksums for unchanged input; platform-reach jobs on the private runner
Evidence from every gate and scan is written to a private evidence store off-platform, so the compliance record survives a full rebuild and independent reviewers never need access to the platform itself.

Invariants

The decisions that keep parts compatible

A small set of binding rules that independently-built units can't read off compliant code — the load-bearing ones:

InvariantWhat it prevents
Code generic, data specificHard-coded deployment facts — one inventory retargets to any hardware
Single network-policy sourceFirewall rules, checks and diagrams silently disagreeing
External control onlyPlatform compromise granting the keys to change the platform
Untrusted build networkBring-up credentials outliving the build — all rotated after cut-over
Three-repo splitDeployment data or evidence leaking into the public pattern repo
Evidence off-platformLosing the compliance record in a reroll; reviewer dependence on the platform

Reroll

Bare metal to healthy, unattended

Disaster recovery and proof-of-quality in one scripted, drilled sequence — assuming only powered machines and an untrusted build network.

1 · installnetwork boot + unattended install
2 · networkfirewall + zones come up
3 · verifyconnectivity proven before cut-over
4 · cut overhosts to the real network; build path retired
5 · rotateevery build-time credential burned
6 · hardenCIS applied + audited
7 · servicesidentity, secrets, mgmt, observability
8 · restoreworkloads + critical data from encrypted backup
≤ 4hbare metal to healthy, incl. data restore
≤ 30 minoperator effort to onboard a service
0manual steps beyond declared power actions
0leaks in the public repo, by review

Destructive operations are double-gated: an explicit confirmation flag and an environment lock that refuses to run against the wrong target, both re-checked at apply-time against a plan-time snapshot.
