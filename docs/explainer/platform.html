On-Prem Dev Platform

A reusable, code-defined blueprint for a small, secure datacentre — one that can be torn down and rebuilt from source in under four hours, proves its own security as it runs, and survives a change of hands. Not one homelab: a pattern any equivalent lab adopts by swapping a single inventory file.

Why

Infrastructure you can throw away

Most home and office labs are hand-built: fragile, undocumented, and lost when the person who built them moves on. This platform inverts that. Everything — network rules, server installs, security hardening, services — is defined as code in version control. The physical hardware is disposable; the code is the asset. The discipline is chosen on greenfield, because it is cheapest to enforce before the first service lands.

A pattern, not a build

A reference rack proves the pattern works. Any equivalent lab stands the same pattern up by swapping inventory — different hardware, same platform — and onboards services one at a time.

Rebuildable from source

The whole environment can be torn down and "rerolled" from code — reproducible and idempotent. Nothing important lives only on a running machine.

Secure by design

Network-segmented by default, hardened to the CIS benchmark before changes can merge, multi-factor for admin access — the security floor is poured before anything stands on it.

Evidenced as it runs

Compliance evidence (UK Secure by Design, CIS, OWASP, MITRE ATT&CK + ATLAS) is produced automatically by the pipeline — never hand-written after the fact.

What it does

Twenty-one capabilities, seven jobs

The specification locks 21 capabilities — each with a stated intent and a concrete success test. Grouped by the job they do. The tags map each group to its underlying capabilities.

Provision & virtualise

Turn bare machines on and install them over the network, unattended — no keyboard, no console — then run everything inside a virtualisation layer with validated network profiles.CAP-1 · 4 · 5 · 6

Network as policy

The firewall is code. The network splits into isolated zones that deny each other by default; only explicitly written rules allow a flow — and every rule is generated from one policy file.CAP-2 · 3

Harden & prove

Servers are hardened to the CIS benchmark before any change can merge, re-scanned on a schedule for drift, and assured against the UK Secure by Design framework with named risk owners.CAP-7 · 10

Identity & access

Multi-factor is mandatory for admin access. Rights follow a lifecycle: recorded approval, single sign-on rolled out in phases, quarterly review, offboarding that provably removes every key.CAP-11 · 16

Secrets & crypto

Secrets are encrypted at rest with defined ownership and leak checks; time sync, DNS validation, internal certificates and at-rest encryption form a baseline every host inherits.CAP-8 · 18

Deliver services safely

Every service onboards through a readiness checklist. Container images are built, scanned, and deployed by digest from a registry the platform pulls — never an inbound push. Intrusion detection stays up even under failover.CAP-12 · 15 · 13

Operate & survive

Automated checks gate every change; backups target a fast rebuild; a signed export can cross an air gap; live state is queryable; a stranger can take over from the docs; and the supply chain is tracked without freezing it.CAP-9 · 14 · 17 · 19 · 20 · 21

The shape · how it works

Six layers, strict rules between them

The platform is built as layers, each depending only on the layers beneath it. Independent pieces stay compatible because the rules between layers are fixed up front — that is what the architecture defines.

L0Power & installturns machines on and installs them over the network, unattended
L1Virtualisationthe hosts running the virtual machines everything lives in
L2Network policythe firewall and network zones — every rule generated from one policy file
L3HardeningCIS security benchmark applied and continuously re-checked for drift
L4Platform servicesidentity & MFA, secrets management, container management, monitoring
L5Workloadsthe actual services, each onboarded through a readiness checklist

Two golden rules: a layer never depends on anything above it (so a rebuild can never deadlock), and logic is generic while specifics live in data files (so moving to different hardware means editing data, not rewriting code).

One consequence worth naming: facts like the network layout exist in exactly one machine-readable file. Firewall rules, virtualisation checks, service placement, and even the diagrams in documents like this one are generated from it — so they cannot quietly disagree.

Rebuild story · how it works

From bare metal to healthy, unattended

"Reroll" is the platform's disaster-recovery plan and its proof of quality: the same scripted sequence, drilled on a schedule. It assumes only powered machines and a basic build network — which is treated as untrusted.

1 · Installmachines boot from the network and install themselves, no keyboard involved
2 · Network upthe firewall and all network zones come up alongside the build network
3 · Verifyconnectivity through the new network is proven before anything is switched over
4 · Cut overhosts move onto the real network; the temporary build path is retired
5 · Burn keysevery credential used during the build is rotated — anything exposed early is now useless
6 · HardenCIS benchmark applied and audited; failure blocks progress
7 · Servicesidentity, secrets, management and monitoring services come up
8 · Restoreworkloads deploy and critical service data is restored from encrypted backup

Destructive steps are double-gated: an explicit confirmation flag and an environment lock that refuses to run against the wrong target. Amber steps are verification gates — the sequence stops rather than proceeds on failure.

Security · how it works

Deny by default, prove continuously

The network is divided into isolated zones. No zone may talk to another unless a written rule allows that specific flow — and those rules are generated from the single policy file, for the main firewall and the standby appliance alike.

Managementadmin access only, MFA enforced
User & developerday-to-day work, no direct admin reach
Servicesplatform and workload services
AI workloadsaccepts only approved API traffic in
Lab / testexperiments, walled off from everything
Untrusted & guestquarantined by default

Zone names above describe roles; the full zone model and per-flow rules live in the policy source. Deployment specifics (addresses, hosts) are intentionally not published.

Hardened, and it stays hardened

CIS benchmark compliance gates every merge, and live hosts are re-scanned on a schedule — drift raises an alert and lands in the evidence record.

Identity first

MFA is mandatory for privileged access. Access follows a lifecycle: recorded approval, quarterly review, and offboarding that provably removes every key — from one owned access file.

Secrets never in the open

All secrets are encrypted at rest in a private repository; automated leak checks and human review guard the public one. The public pattern never contains real addresses or credentials.

Failover without going blind

If the main firewall fails, a standby appliance holds the same deny-by-default policy. Entering failover raises an alert and restoring full inspection is tracked to completion.

Risks & boundaries

What we're honest about

A plan that hides its risks isn't credible, and a scope with no edges grows without limit. Both are stated plainly and tracked, not wished away.

ItemPosition
Young management tool watchedThe container-management service is a young project with a recent serious vulnerability. Mitigations: minimum safe version enforced, isolated management zone, fast security-update policy, and a named fallback product with a plain exit path.
Ageing standby firewall watchedThe failover appliance is older hardware. It only carries the continuity role, holds the same policy, and its replacement is an inventory swap by design.
Single administrator mitigatedHandover survivability is a tested requirement: documentation drills, a sealed recovery seed, and a disposable admin workstation mean a stranger can take over.
Deepest hardening level deferredCIS Level 1 is the enforced gate today; Level 2 on the hypervisor is a 12-month target pending validation on live hardware, with formal waivers where controls conflict.
Monitoring product choice deferred safelyThe rules are fixed (central logs, 30-day/1-year retention, a deliberately short alert list, log archive survives rebuilds); the specific product is chosen at implementation.
Explicit non-goals keep the scope honest: no bulk service rollout yet, no offsite backup until capacity exists, no host full-disk encryption (it conflicts with unattended rebuild), no forensic archive through a reroll, and no lock to specific hardware models. Each is a chosen exclusion with a reason, not an oversight.
